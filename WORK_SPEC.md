# 高考物理交互图审核工作规范

本文件约束本仓库后续所有交互图审核、修复、提交与进度汇报。

## 1. 工作目标

逐页验证每个交互图在桌面端和 390px 手机端均可实际使用。页面只有在下列项目全部成立时，才可记录为通过：

1. 每个可见按钮都能改变对应的物理状态、画面或关键读数。
2. 播放、暂停、重置具有真实的物理过程和可观察结果。
3. 至少一个直接操作或参数输入能联动画面、读数与公式。
4. 公式中的物理量可在画面、仪器读数、轨迹或图像中对应。
5. 手机端无横向溢出，控件可见、可点击、可触摸。

自动检查通过不是教学质量的唯一证据；发现物理错误、无意义动画或公式与画面脱节时，必须修复并重新审核。

## 2. 连续执行规则

- 每完成一个审核批次，立即启动下一批；提交和汇报都不是停止开发的节点。
- 只有以下情况允许暂停：用户明确要求暂停、遇到不可自行解决的外部阻塞、或全部页面完成。
- 每页记录开始时间、结束时间、审核结果和修复说明。
- 单页超过 30 分钟仍未完成时，立即进行阻塞诊断，记录具体原因、已排查项和下一步；不得静默等待。

## 3. 30 分钟进度汇报

每 30 分钟向用户报告一次真实状态：

- 本时段已完成的页面与结果；
- 当前正在处理的页面；
- 最近一次真实 GitHub 提交（哈希和时间）；
- 若本时段没有可交付结果，说明原因、排查情况和下一步动作。

禁止把本地未提交改动、旧提交或自动审计分数表述为“刚刚上传”或“已完成开发”。

## 4. 提交与推送

- 每累计 20 个独立页面完成审核与必要修复后，运行严格检查、提交并推送当前 Pages 分支。
- 提交前必须核对 `git status`；只暂存本批次明确归属的文件。
- 不得混入未归属的题目导入、资料整理或其他用户改动。
- 推送后必须用远端跟踪状态和最新提交哈希确认成功，才可报告“已上传”。

## 5. 审核记录

- 每批次保存可追溯的审核结果，注明工具检查范围和人工复核范围。
- 页面失败时，记录失败按钮/参数、预期物理行为、实际行为、修复内容和复测结果。
- 不得以“按钮文字变化”“时钟变化”作为物理过程已发生的唯一依据。

## 6. 完成条件

所有交互图均已逐页审核；每个失败项已修复并复测；最后一次严格检查通过；全部审核记录和源代码均已真实推送到 GitHub Pages 分支。
