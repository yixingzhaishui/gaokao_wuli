# 高考物理交互图审核工作规范

> **2026-07-15 内容审核覆盖规则：**历史题数、覆盖率、交互通过和 `DONE` 记录只作为旧阶段证据，不再推导为“已核验题库”或“可正式发布”。来源发布必须符合 `SOURCE_POLICY.md` 2.0；内容发布必须通过 `data/progress.json` 的全部审核轴。OpenAI 内容审核报告中的 P0/P1 要求优先于本文件的历史口径。

本文件约束本仓库后续所有交互图审核、修复、提交与进度汇报。

## 1. 工作目标

逐页验证每个交互图在桌面端和 390px 手机端均可实际使用。页面只有在下列项目全部成立时，才可记录为通过：

1. 每个可见按钮都能改变对应的物理状态、画面或关键读数。
2. 播放、暂停、重置具有真实的物理过程和可观察结果。
3. 至少一个直接操作或参数输入能联动画面、读数与公式。
4. 公式中的物理量可在画面、仪器读数、轨迹或图像中对应。
5. 手机端无横向溢出，控件可见、可点击、可触摸。

自动检查通过不是教学质量的唯一证据；发现物理错误、无意义动画或公式与画面脱节时，必须修复并重新审核。

## 2. 连续执行规则

- 每完成一个审核批次，立即启动下一批；提交和汇报都不是停止开发的节点。
- 只有以下情况允许暂停：用户明确要求暂停、遇到不可自行解决的外部阻塞、或全部页面完成。
- 每页记录开始时间、结束时间、审核结果和修复说明。
- 单页超过 30 分钟仍未完成时，立即进行阻塞诊断，记录具体原因、已排查项和下一步；不得静默等待。

## 3. 30 分钟进度汇报

每 30 分钟向用户报告一次真实状态：

- 本时段已完成的页面与结果；
- 当前正在处理的页面；
- 最近一次真实 GitHub 提交（哈希和时间）；
- 若本时段没有可交付结果，说明原因、排查情况和下一步动作。

禁止把本地未提交改动、旧提交或自动审计分数表述为“刚刚上传”或“已完成开发”。

## 4. 提交与推送

- 每累计 20 个独立页面完成审核与必要修复后，运行严格检查、提交并推送当前 Pages 分支。
- 提交前必须核对 `git status`；只暂存本批次明确归属的文件。
- 不得混入未归属的题目导入、资料整理或其他用户改动。
- 推送后必须用远端跟踪状态和最新提交哈希确认成功，才可报告“已上传”。

## 5. 审核记录

- 每批次保存可追溯的审核结果，注明工具检查范围和人工复核范围。
- 页面失败时，记录失败按钮/参数、预期物理行为、实际行为、修复内容和复测结果。
- 不得以“按钮文字变化”“时钟变化”作为物理过程已发生的唯一依据。

## 6. 完成条件

所有交互图均已逐页审核；每个失败项已修复并复测；最后一次严格检查通过；全部审核记录和源代码均已真实推送到 GitHub Pages 分支。

# 真题/可靠模拟题修复 Task Spec

本节约束当前“真题、例题审核与可靠来源补题”任务。若本节与上方交互图审核规范发生冲突，当前任务以本节为准。

## A. 当前任务目标

把学生端可见例题替换、补充为来源可靠、可追溯、已核验的高考真题或可靠模拟题，逐步解决知识点缺少真题/例题的问题。

- 单批导入文件：`data/import-batches/*.json`
- 进度监控：`scripts/progress-monitor.js`
- 严格检查：`scripts/check.js`

## B. 来源硬规则

1. 优先使用官方高考真题、官方答案、权威教育考试机构材料。
2. 可使用可靠模拟联考题，但必须有可追溯来源，例如原卷 PDF、答案 PDF、学校/机构正式发布页。
3. 不允许编造题、不允许无来源改编题冒充真题。
4. 若题目来自模拟题，`source_kind`、`source_detail`、`source_urls` 必须明确说明其模拟来源。
5. 每道题必须写入：
   - `source`
   - `source_kind`
   - `source_verified`
   - `source_detail`
   - `source_urls`
   - `verification`
   - `verified_at`
6. 只有题干、选项、答案、解析逻辑均可核验时，才允许进入 `data/verified-problems.json`。
7. 旧题库、页面旧例题、自动补充题在未核验前继续隔离，不进入学生端。

## C. 单题处理流程

每完成 1 道题，必须按以下顺序执行，不得跳步：

1. 读取原卷/答案来源，确认题干、选项、答案可追溯。
2. 独立核验答案，不只照抄答案页。
3. 创建或更新对应 `data/import-batches/*.json`。
4. 运行干跑：
   - `node scripts/import-verified-batch.js <batch-file> --dry-run`
5. 正式导入：
   - `node scripts/import-verified-batch.js <batch-file> --apply`
6. 记录进度：
   - `node scripts/progress-monitor.js mark "imported <problem-id>"`
7. 运行严格检查：
   - `node scripts/check.js`
8. 向用户汇报该题完成状态、学生端题数、覆盖数、检查结果。

## D. 进度汇报规则

1. 每完成 1 个题目任务点，立即汇报一次。
2. 每 30 分钟必须汇报一次真实进度，即使没有新增题。
3. 30 分钟汇报必须包含：
   - 学生端已核验题数
   - 知识点覆盖数
   - 最近完成题目 ID
   - `scripts/check.js` 是否通过
   - Git 是否有文件变更
   - 若无新增，说明原因和下一步
4. Heartbeat/自动提醒只是触发器，不是暂停开发的理由。
5. 建立了 30 分钟 heartbeat 后，仍必须继续主动执行“找题 → 核验 → 导入 → 检查”的流水线。

## E. 停止与阻塞规则

除非出现以下情况，否则不得停止开发：

1. 用户明确要求暂停。
2. 遇到无法自行解决的外部阻塞，例如来源不可访问且没有可替代可靠来源。
3. 需要破坏性操作或不可逆操作。
4. 全部任务完成。

如果 30 分钟内没有新增学生端题数，必须立即执行阻塞诊断：

1. 运行 `node scripts/progress-monitor.js check`。
2. 运行 `node scripts/check.js`。
3. 检查 `git status --short`。
4. 明确报告没有进展的原因。
5. 继续选择下一道可靠来源题推进。

## F. 当前阶段完成标准

当前阶段不以“达到某个整数题数”作为停止点。达到 200、220、240 等节点后，仍需继续推进，直到：

1. 所有可取得可靠来源的缺失知识点都至少有可展示真题或可靠模拟题；
2. 无可靠来源的缺口被明确记录为“缺来源”，并说明查找路径；
3. `node scripts/check.js` 通过；
4. Git 工作区变更可被用户在 GitHub Desktop 看到；
5. 用户确认可以停止。
